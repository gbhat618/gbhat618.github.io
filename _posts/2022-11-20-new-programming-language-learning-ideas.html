When we already have experience in one programming language like Python or Java by writing software for a few years, learning a new language should become easier.

However several questions arise,

  • If I want to learn a programming language, how do I measure my success levels?

  • One can get to too low level in programming languages, so how much should I learn? For an application engineer, knowing how to develop a modular application relatively quickly, is probably the benchmark

  • Then, what does it take to be able to write a good modular application?

This blog post contains some notes along this thought process...

Level 1: Basic constructs

Knowing the syntax of the language, data types, loops, if-else statements, writing methods, and classes and how to run the code, built-in data structures - more or less a ++ version of “Hello World”.

With this one can do problem-solving for algorithm data structures, probably pass Level 1 interviews. After doing some online tutorials etc, basic programming, and problem-solving this can be achieved.

Level 2: Simple scripts or small APIs

Knowing how to make work with data exchange between,

  • file systems

  • databases

  • network calls

  • multi-threading and multiprocessing

The idea here is, that the code is not just a stdin → stdout, or in-memory based problem solving, but interacts with other software interfaces.

Some simple exercises can be,

  • Reading and writing to files in different formats JSON/CSV etc

  • Making network calls to GET/POST etc

  • Making simple SQL queries to the databases for CRUD operations

This is for being able to get some automation tasks done by using the programming language. At this point, we can start seeing some clear returns for putting in the time to learn the language.

Level 3: Back to Basics

After the clearing Level 2 exercise, we should try to cover our blind spots and bridge the knowledge gaps.

Maybe it is only possible by reading a book on “language reference”.  The language reference books should cover pretty much everything that the language has to offer with clear code examples and discussions,

For example,

  • Syntax, data types, default values, etc

  • Modules and packages: Structuring constructs for hundreds of lines of code

Ex:

  • python import, __init__ files, pythonpath, etc

  • Java import, module, package, access specifiers, etc

  • Multithreading and multiprocessing

  • Network APIs

  • File APIs

  • Other Core features of the language

  • Built-in data structures

It will make things crystal clear as the past experience gets reinforced and several questions about the language get answered.

Example books,

  • Java: The Complete Reference

  • The Go Programming Language

  • C++: The Complete Reference

Level 3a:  Idioms of the language

Just like natural languages, programming languages have their own idioms or best practices to do specific things, like creating a hashmap from a list, filtering a list of items, and a lot of other things.

This is where sayings such as “Pythonic way of doing” or “the Java way of doing” etc come in.

I think this can be learned by following some really good blogs, and articles. There are would some books as well. A book would be preferred to cover a lot of things in a short time or to use as a handbook guide to refer to from time to time.

Example books,

  • Python Tricks: A Buffet Of Awesome Python Features

  • Python cookbook

  • Effective Java

Level 3b: Tooling, packaging system, and common libraries

So far all these things above are writing programs/scripts that are expected to be used by the authors themselves, but to scale it to a professional developer level and work on large projects, one needs to know the tools. Some of these tools are transferable across languages, while some are language ecosystem specific.

  1. Common toolings: git, GitHub, debugging basics, IDEs, documentation reading, etc.

  2. Specific toolings: build tools, package repos, package installation, package paths, etc.

    1. Java: maven, gradle, maven central repo, etc.

    2. Python: pypy, pip, etc.

  3. Common libraries: every language has some de facto library for certain functionalities, so getting a good understanding of their APIs would be good.

  1. Java: Apache Commons, Google Guava, Jackson, args4j, etc

  2. Python: requests, csv, json, pandas, etc.

The best to learn this is to work on at least 3-4 real-world projects.

Level 4: Frameworks that are platforms

There are some frameworks and tools that would have become like a platform to be applicable for certain job roles, where these developers spend the majority of their time in these specific frameworks/platforms. This would include the ORM layers, the API layer for web apps, specific CLI frameworks, and several more.

The relevance of these tools is Job Profile Specific. And would be a learning-as-you-go kind, but doing some practice and courses on them is really helpful, and really pays off on the job.

For example,

  1. Java: Spring framework, spring boot, hibernate, quarkus, micronaught, etc

  2. Python: flask, SQLalchemy, Django, pandas, NumPy, TensorFlow, etc. the jupyter notebook platform, etc.

Level 5: Profilers, benchmarking, debugging, and performance best practices

This section is more on the senior-level developers who are deep into software development in that specific language.

At some point, there will be performance and memory issues, so need to be aware of the tools and how to measure things in that language. The general ideas of performance best practices are relevant for all languages, but the tools are mostly specific to each language.

  • Java: jstack, jmap, eclipse mat tool, jprofiler, signal handling, etc.

  • Python: cProfile, kCacheGrind, snakeviz, etc

There is probably more need to be thought about on this part...

Level 6: Internals of the language, memory models, and concurrency

  • The memory model of the language

  • Garbage collection or memory management APIs

  • Concurrency patterns, threads, virtual or green threads, and coroutines, processes. Interthread and inter-process communications.

  • Thread-Process lifecycle, performance, and memory cost, etc for threads and processes.
